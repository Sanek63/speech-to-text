# Транскрипция + диаризация с определением роли (Преподаватель / Студент)

Пайплайн, который берёт запись занятия (лекция, семинар, защита) и выдаёт транскрипт, где каждая реплика размечена не просто меткой спикера (`SPEAKER_00`), а ролью — `Преподаватель` / `Студент`.

## Технологии

| Задача | Технология | Почему |
|---|---|---|
| ASR (речь → текст) | [`openai-whisper`](https://github.com/openai/whisper), модель `large-v3` | Лучший WER для русского среди открытых моделей; встроенные word-level таймкоды |
| Диаризация (кто говорит) | [NVIDIA NeMo](https://github.com/NVIDIA/NeMo) MSDD (`diar_msdd_telephonic`) | Качество в одном классе с `pyannote`, включая детекцию перекрывающейся речи |
| Резка аудио / формат | `ffmpeg` / `ffprobe` | Конвертация в 16kHz mono WAV, `silencedetect` для чанкования, вырезка кусков |
| Ролевая классификация | Собственная rule-based эвристика (без ML) | Явные, объяснимые правила — не чёрный ящик, не нужны размеченные данные |
| Оценка качества | `pymupdf` (PyMuPDF) + `difflib` | Разбор эталонного PDF, текстовое выравнивание для WER и accuracy ролей |
| Работа с конфигами NeMo | `omegaconf` | Формат конфига, который использует сам NeMo |

## Из чего состоит пайплайн

```
audio.mp3
   │
   ▼
1. preprocess_audio()        ffmpeg → 16kHz mono WAV (+ loudness normalize)
   │
   ▼
2. (опционально) split_audio_on_silence()   резка на куски по паузам — см. "Чанкование"
   │
   ▼
3. transcribe_with_model() / transcribe_chunked()  ASR: текст + таймкоды на уровне слова
   │
   ▼
4. diarize_nemo()             NeMo MSDD: временные отрезки → анонимные SPEAKER_XX
   │
   ▼
5. assign_word_speakers()     каждому слову присваивается спикер по максимальному
   │                          перекрытию с диаризационным отрезком
   ▼
6. group_into_turns()         подряд идущие слова одного спикера → реплики (Turn)
   │
   ▼
7. extract_role_features()    признаки по каждому спикеру (см. таблицу ниже)
   │
   ▼
8. assign_roles()             признаки → взвешенный score → Преподаватель / Студент(ы) / Other
   │
   ▼
9. export_text/srt/json()     три формата вывода
   │
   ▼
10. evaluate_against_pdf()    (опционально) WER + accuracy ролей против audio/script.pdf
```

ASR и диаризация — полностью независимые модели: Whisper ничего не знает про спикеров (только звук → буквы), NeMo ничего не знает про текст (только звук → анонимные кластеры голоса по тембру). Шаг 5 — единственное место, где они склеиваются: слову присваивается тот диаризационный отрезок, который перекрывает его по времени сильнее всего (если слово попало точно в паузу между отрезками — берётся ближайший по времени).

## Как определяется роль (`assign_roles`, `extract_role_features`)

Наивный подход «кто говорит больше — тот преподаватель» не работает: в записи-эталоне (`audio/script.pdf`) докладчик, размеченный как `[Студент]`, говорит 70–80% времени одним монологом, а модератор (`[Преподаватель]`) — всего несколько реплик. Поэтому вместо одного признака — взвешенный скоринг по нескольким:

| Признак | Вес | Смысл |
|---|---|---|
| `turn_count` (число реплик) | 0.25 | Модератор берёт слово часто и коротко (направляет, уточняет); докладчик — редко, но длинными монологами |
| `question_density` (вопросов на 100 слов) | 0.25 | Преподаватель/экзаменатор задаёт больше вопросов относительно объёма своей речи, даже если говорит меньше |
| `avg_turn_duration`⁻¹ (обратная средняя длина реплики) | 0.15 | Короткие частые реплики = направляющая роль |
| `is_first` (открыл сессию) | 0.15 | Модератор почти всегда открывает занятие |
| `is_last` (закрыл сессию) | 0.10 | И почти всегда закрывает |
| `marker_score` (лексические маркеры, см. ниже) | 0.10 | Прямые текстовые сигналы роли |

Лексические маркеры — счётчик вхождений в текст реплик спикера (`extract_role_features`):

- **Маркеры преподавателя**: `давайте`, `переходим к`, `на этом закончим`, `спасибо за доклад`, `коллеги`, `следующий вопрос`, `оппонент`, `комиссия`, `прошу вопросы`, `передаю слово`, `представьтесь`, `у нас есть время`
- **Маркеры студента**: `я исследовал`, `в своей работе`, `целью моей работы было`, `хотел бы представить`, `моя работа посвящена`, `в результате исследования`, `мой доклад`, `я хочу рассказать`
- **Маркеры вопроса** (для `question_density`): `?`, `почему`, `как вы думаете`, `поясните`, `объясните`, `согласны ли`, `в чём`, `что если`, `каким образом`, `расскажите`

Алгоритм (`assign_roles`):
1. **Фильтрация мелких кластеров** — спикеры с суммарной речью <3% от общего времени (`MIN_SIGNIFICANT_DURATION_RATIO`) уходят в `Other` (фоновая болтовня зала) и не участвуют в голосовании.
2. **Нормализация** — каждый признак приводится к [0, 1] относительно min/max среди оставшихся спикеров (не абсолютные пороги: ролевое поведение относительно, обучающих данных нет).
3. **Взвешенная сумма** → `teacher_score` на каждого спикера.
4. **Разбиение по разрыву в score** (`_split_by_score_gap`) — спикеры сортируются по убыванию score, ищется наибольший разрыв между соседними значениями; всё, что выше разрыва → `Преподаватель`(-и), ниже → `Студент`(-ы). Это, а не жёсткое «ровно один преподаватель», покрывает кейс комиссии из нескольких преподавателей на защите: `Преподаватель 1`, `Преподаватель 2`, ... При нескольких студентах — аналогично, `Студент 1`, `Студент 2`.
5. Если значимый спикер всего один — роль помечается как `Неизвестно (только один значимый спикер)`, а не угадывается вслепую.

Веса и пороги — константы в `app/pipeline/config.py` (`ROLE_WEIGHTS`, `MIN_SIGNIFICANT_DURATION_RATIO`), подобраны вручную и легко перенастраиваются без изменения логики.

## Чанкование

Резать приходится только ASR — это стадия с наибольшим потреблением GPU-памяти на длинных записях. **Диаризация NeMo всегда идёт одним проходом на весь файл**, чанкованием не затрагивается:

- NeMo сама справляется с длинными записями через внутреннее скользящее окно (`diar_window_length` в её конфиге) — chunking + пересборка идентичности спикера между кусками (самая рискованная часть в других реализациях) здесь просто не нужна.
- Поскольку диаризация не чанкуется, спикер каждому слову назначается по единой глобальной разметке независимо от того, как порезан ASR — `group_into_turns` просто сравнивает спикера соседних слов, ему всё равно, откуда они взялись. Это защищает признак `turn_count`: длинный монолог, случайно разрезанный по границе ASR-чанка, всё равно останется одной репликой.

Как режется ASR (`split_audio_on_silence`, включается флагом `--chunked` / `CHUNK_LONG_AUDIO = True` в ноутбуке):
1. `ffmpeg -af silencedetect` находит паузы в аудио.
2. Целевая длина куска — `CHUNK_TARGET_SEC` (600с по умолчанию), точка разреза ищется как пауза, ближайшая к цели, в окне `[target×0.5, max_sec]`.
3. Если пауза не находится до `CHUNK_MAX_SEC` (720с) — режется жёстко на границе (единственный оправданный случай: сплошной монолог длиннее лимита, как раз наш тестовый докладчик).
4. Каждый кусок транскрибируется той же загруженной моделью Whisper (`transcribe_chunked`), с `initial_prompt` = хвост текста предыдущего куска — для связности терминологии и имён собственных между кусками.
5. Таймкоды слов/сегментов сдвигаются на абсолютный offset куска — на выходе тот же формат, что и без чанкования, весь downstream-код не меняется.

Включается по-разному в зависимости от точки входа: в CLI/ноутбуке — вручную (`--chunked` / `CHUNK_LONG_AUDIO = True`, по умолчанию выключено — тестовая запись ~30 мин влезает в память GPU целиком). В backend (`app/main.py`) — **автоматически**, по длительности файла (`CHUNK_THRESHOLD_SEC`, порог 45 минут), поскольку через UI флаг всё равно не выставить.

| H2 | **Объём речи — недостаточный признак роли «преподаватель»**: в формате защиты/гостевого доклада студент/докладчик может говорить бóльшую часть времени | Разбор `audio/script.pdf`: докладчик (`[Студент]`) говорит 70–80% времени одним монологом, модератор (`[Преподаватель]`) — 4 реплики | **Подтверждена** на тестовом файле — повод для всей архитектуры Stage 5 |

| H4 | Диаризацию не обязательно чанковать — она сама переваривает длинную запись за один проход | Осознанно не реализована кросс-чанковая пересборка идентичности спикера; чанкуется только ASR | Архитектурное решение, держит `turn_count` корректным даже при разрезании длинного монолога |

## Запуск

Три равноценных способа, для разных сценариев:

**CLI** (быстрый разовый прогон, требует локальную GPU-машину):
```bash
cd app
pip install -r requirements.txt
python -m pipeline.cli ../audio/audio.mp3 --reference ../audio/script.pdf

# для очень длинных записей (>45-60 мин):
python -m pipeline.cli ../audio/audio.mp3 --chunked --chunk-target-sec 600 --chunk-max-sec 720
```
Дорогие стадии (ASR + диаризация) кэшируются в `<outdir>/.cache/` — повторный запуск (например, после падения на экспорте) не пересчитывает их заново; `--force` сбрасывает кэш.

**Colab** — открыть `pipeline_nemo.ipynb` (полностью самодостаточный, не зависит от `app/`), включить GPU-рантайм (Runtime → Change runtime type → GPU) и выполнять ячейки по порядку.

**Backend + UI** — см. следующий раздел.

## Backend + UI (`app/`)

Тот же пайплайн (декомпозированный в пакет `app/pipeline/`, см. `app/pipeline/__init__.py` для публичного API), но как долгоживущий сервис: FastAPI держит модели прогретыми, отдаёт REST API и минималистичный веб-плеер с диаризованным транскриптом (клик по реплике — перемотка аудио, live-подсветка текущего слова, RTF-метрики по каждой стадии).

Перед первым запуском стоит проверить, что GPU реально проброшена в Docker (не вопрос производительности — если `nvidia-container-toolkit` не настроен, GPU в контейнере просто не будет видна):
```bash
docker run --rm --gpus all nvidia/cuda:12.1.0-base-ubuntu22.04 nvidia-smi
```

```bash
cd app
docker-compose up --build
# открыть http://<host>:11111
```

API: `POST /api/upload` (файл → `file_id`) → `POST /api/transcribe/{file_id}` (ставит в очередь) → опрос `GET /api/status/{file_id}` (стадия/RTF-метрики) → `GET /api/transcript/{file_id}`. ASR и диаризация внутри одной задачи никогда не выполняются параллельно (см. "Жизненный цикл моделей" в коде `app/main.py`) — на одной GPU-карте это уже приводило к OOM; несколько задач при этом можно ставить в очередь одновременно, не блокируя загрузку/опрос статуса.

## Формат вывода

Три формата из общей структуры `Turn(speaker, role, start, end, text, words)`:
- `transcript.txt` — текст с ролями, напрямую сравним с `script.pdf`
- `transcript.srt` — субтитры с меткой роли на каждый кадр (длинные реплики режутся на кадры по 14 слов — иначе многоминутный монолог превращается в один нечитаемый SRT-кадр)
- `transcript.json` — под будущий плеер (перемотка + синхронный транскрипт), см. ниже

### `transcript.json` — годится ли для плеера с перемоткой и синхронным транскриптом

Да, схема рассчитана именно на это: `start`/`end` — секунды, то же, что `audio.currentTime` у HTML5 `<audio>`/`<video>`, конвертация не нужна.

```json
{
  "audio_file": "audio.mp3",
  "language": "ru",
  "duration": 1798.2,
  "turns": [
    {
      "id": 0,
      "speaker": "SPEAKER_00",
      "role": "Преподаватель",
      "start": 0.0,
      "end": 5.2,
      "text": "Давайте начнём, коллеги",
      "words": [
        {"word": "Давайте", "start": 0.0, "end": 0.4, "confidence": 0.97},
        {"word": "начнём", "start": 0.5, "end": 0.9, "confidence": 0.95}
      ]
    }
  ]
}
```

Как это ложится на описанный сценарий (плеер снизу — транскрипт с таймлайном, клик по фразе — перемотка, во время воспроизведения — подсветка/переключение текущей фразы):
- **Список реплик снизу с таймлайном** — рендерится прямо из `turns[]`, `id` — стабильный ключ для React/Vue-списка.
- **Клик по фразе → перемотка аудио** — `audioElement.currentTime = turn.start`.
- **Подсветка/автопереключение текущей фразы при воспроизведении** — на `timeupdate` ищете реплику, где `turn.start <= currentTime < turn.end` (или конкретное слово внутри `turn.words` — то же самое, но на уровень глубже, для karaoke-подсветки слова, а не всей реплики).
- **`confidence`** (0–1, из Whisper) — не обязателен для базового сценария, но полезен фронту, чтобы визуально пометить неуверенно распознанные слова (например, подчеркнуть).

Статус обработки ("в очереди/обрабатывается/готово") с тех пор реализован — см. `GET /api/status/{file_id}` в разделе Backend + UI выше. Чего всё ещё нет и осознанно не строилось (не требовалось для демо): пользователей/авторизации и версионирования транскрипта при повторном пересчёте (`/api/transcribe` на тот же `file_id` просто перезапишет `transcript.json`).

## Известные ограничения

- Роль — ручная эвристика, а не обученная модель: веса скоринга подобраны вручную, не откалиброваны на датасете.
- NeMo не поддерживает Apple MPS (только CUDA/CPU) — на Mac без внешней NVIDIA-карты будет работать медленно на CPU; для Colab это не проблема.
- `NEMO_DIAR_CONFIG_URL` зафиксирован под `nemo_toolkit==2.7.3` — при другой версии пакета схема YAML-конфига диаризации может не совпасть (сама диаризация в NeMo — CUDA-first и не поддерживает Apple MPS).
- Diarization всегда одним проходом на весь файл — при очень длинной записи, не влезающей в память GPU целиком, это станет следующим узким местом (ASR к тому моменту уже решается чанкованием).
