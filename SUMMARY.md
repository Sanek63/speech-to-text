# Краткая сводка проекта: технологии, пайплайн, корнер-кейсы

Сжатая техническая сводка по текущему состоянию проекта — для доклада и быстрого онбординга. Полное описание со структурой проекта, гипотезами и деталями запуска — в [README.md](README.md).

Суть сервиса: берёт запись занятия (лекция, семинар, защита) и отдаёт транскрипт, где каждая реплика размечена не просто меткой спикера (`SPEAKER_00`), а ролью — `Преподаватель` / `Студент`. Загрузка через веб-UI или API, обработка асинхронная (очередь + фон), результат — плеер с синхронным транскриптом.

## Технологии

| Задача | Технология | Почему |
|---|---|---|
| ASR (речь → текст) | Веса `large-v3` [openai-whisper](https://github.com/openai/whisper), инференс — [faster-whisper](https://github.com/SYSTRAN/faster-whisper) (CTranslate2 `4.8.1`) | Лучший WER для русского среди открытых моделей; встроенные word-level таймкоды. CTranslate2 ощутимо быстрее и экономнее по GPU-памяти, чем эталонный PyTorch-рантайм openai-whisper — конвертация весов openai→HF→CT2 полностью локальная (`pipeline/ct2_convert.py`), ни разу не обращается к huggingface.co |
| Диаризация (кто говорит) | [NVIDIA NeMo](https://github.com/NVIDIA/NeMo) `2.6.0`, MSDD (`diar_msdd_telephonic`) | VAD + speaker embeddings + нейросетевой диаризатор в одном пайплайне; качество в одном классе с `pyannote`, веса — с NGC, не с HuggingFace |
| Резка аудио / формат | `ffmpeg` / `ffprobe` | Конвертация в 16kHz mono WAV, извлечение аудиодорожки из видео (`-vn`), `silencedetect` для чанкования ASR |
| Ролевая классификация | Собственная rule-based эвристика (без ML) | Явные, объяснимые правила, не чёрный ящик — датасета для обучения классификатора нет |
| Backend / очередь | FastAPI + Postgres (источник правды по задачам) + Redis (очередь) | `backend` (лёгкий, без CUDA) и `worker` (тяжёлый, GPU) — раздельные контейнеры, общаются только через Postgres/Redis |
| Фронтенд | Vanilla HTML/JS/CSS, без сборки | Список задач → отдельная страница на каждую задачу: плеер, метрики, транскрипт с автоподсветкой текущей реплики/слова |

Версии `torch==2.4.0` / `transformers~=4.53.0` / `ctranslate2==4.8.1` / `nemo_toolkit[asr]==2.6.0` — не независимый выбор каждой библиотеки, а единственный найденный рабочий срез: `nemo_toolkit` тянет конкретный `transformers`, тот должен быть достаточно старым, чтобы не требовать `torch>=2.6.0` (иначе конфликт с `nemo_toolkit`), а `ctranslate2` должен быть именно `4.8.1`, не старше (у `4.4.0` в рантайме на образе с cuDNN9 не хватает `libcudnn_ops_infer.so.8`) — см. комментарии в `requirements-worker.txt`.

## Как проходит обработка

```
audio/video
   │
   ▼
1. preprocess_audio()        ffmpeg → 16kHz mono WAV (+ loudness normalize; видео → только звук)
   │
   ▼
2. (если запись длинная) split_audio_on_silence()   резка ASR на куски — см. "Чанкование"
   │
   ▼
3. transcribe_with_model() / transcribe_chunked()    faster-whisper: текст + таймкоды на уровне слова
   │
   ▼
4. diarize_nemo()             NeMo MSDD: временные отрезки → анонимные SPEAKER_XX (всегда одним проходом, без чанкования)
   │
   ▼
5. assign_word_speakers()     каждому слову — спикер по максимальному перекрытию с диаризационным отрезком
   │
   ▼
6. group_into_turns()         подряд идущие слова одного спикера → реплики
   │
   ▼
7. extract_role_features() → assign_roles()     признаки поведения спикера → Преподаватель / Студент(ы) / Other
   │
   ▼
8. export + запись в Postgres     transcript.json с ролями, метрики (время/RTF по стадиям)
```

ASR и диаризация — независимые модели: Whisper не знает о спикерах (только звук → буквы), NeMo не знает о тексте (только звук → анонимные кластеры голоса по тембру). Шаг 5 — единственное место склейки.

### Как определяется роль

Наивный признак «кто говорит больше — тот преподаватель» не работает: на защите докладчик (студент) может говорить 70–80% времени одним монологом, а экзаменатор — всего несколько реплик. Вместо этого — взвешенный скоринг по поведению:

| Признак | Вес | Смысл |
|---|---|---|
| `turn_count` (число реплик) | 0.25 | Модератор берёт слово часто и коротко; докладчик — редко, но длинными монологами |
| `question_density` | 0.25 | Экзаменатор задаёт больше вопросов относительно объёма своей речи |
| `avg_turn_duration`⁻¹ | 0.15 | Короткие частые реплики = направляющая роль |
| `is_first` / `is_last` | 0.15 / 0.10 | Модератор почти всегда открывает и закрывает занятие |
| `marker_score` (лексические маркеры) | 0.10 | Прямые текстовые сигналы: `«давайте»`, `«коллеги»`, `«оппонент»` → преподаватель; `«в своей работе»`, `«целью моей работы было»` → студент |

Спикеры с <3% от общей длительности уходят в `Other` (не участвуют в голосовании). Признаки нормализуются в [0, 1] относительно min/max среди оставшихся спикеров (не абсолютные пороги — ролевое поведение относительно). Разбиение на роли — по наибольшему разрыву в итоговом score между соседними по рангу спикерами (`_split_by_score_gap`), а не жёсткое «один преподаватель» — это покрывает и комиссию из нескольких преподавателей, и нескольких студентов сразу. Веса — константы в `app/pipeline/config.py`.

### Чанкование

Режется только ASR — самая GPU-память-затратная стадия на длинных записях. **Диаризация всегда идёт одним проходом на весь файл**: NeMo сама справляется с длинными записями через внутреннее скользящее окно, а раз она не чанкуется — спикер каждому слову назначается по единой глобальной разметке независимо от того, как порезан ASR. Это защищает `turn_count`: длинный монолог, случайно разрезанный по границе ASR-чанка, всё равно остаётся одной репликой.

Как режется ASR (`split_audio_on_silence`, включается автоматически при длительности файла >45 минут):
1. `ffmpeg -af silencedetect` находит паузы.
2. Целевая длина куска — 600с, точка разреза — ближайшая к цели пауза.
3. Если пауза не находится до 720с — режется жёстко на границе (сплошной монолог без пауз длиннее лимита).
4. Каждый кусок транскрибируется одной и той же загруженной моделью, с `initial_prompt` = хвост текста предыдущего куска — для связности терминологии и имён собственных между кусками.
5. Таймкоды сдвигаются на абсолютный offset куска — downstream-код не отличает чанкованный путь от обычного.

## Корнер-кейсы и как они решены

| Кейс | Проблема | Решение |
|---|---|---|
| Роль ≠ объём речи | Докладчик на защите говорит большую часть времени, но не он ведёт занятие | Ролевой скоринг по поведению (см. выше), а не по длительности речи |
| Нет доступа к huggingface.co | Стандартный стек для задачи — `whisperx` + `pyannote.audio` — тянет веса именно оттуда; сеть, где развёрнут сервис, домен полностью блокирует | Модели, которые качаются напрямую от вендора: openai-whisper — с CDN OpenAI, NeMo — с NGC. Конвертация openai-whisper → HF → CTranslate2 (ради скорости) сделана полностью локально, без обращения к HuggingFace ни на одном шаге (`pipeline/ct2_convert.py`, `pipeline/_openai_to_hf.py`) |
| Whisper + NeMo не помещаются в GPU одновременно | OOM при одновременной резидентности двух моделей на карте | `worker.py` держит только одну модель на GPU в моменте: Whisper выгружается на CPU (`unload_model(to_cpu=True)`) перед стадией диаризации, NeMo — перед следующей ASR-задачей |
| Самозацикливание текста на длинных записях | После миграции на faster-whisper: под конец многочасовых записей один и тот же абзац повторялся по 10–15 раз подряд | По умолчанию `condition_on_previous_text=True` — текст каждого ~30-секундного окна декодирования попадает в промпт следующего; одна галлюцинированная, но правдоподобная фраза запускает самоусиливающуюся петлю. Каждое повторение — чистый беглый сегмент, поэтому `compression_ratio_threshold` его не ловит (оценивает один сегмент, не повтор между сегментами). Фикс — `condition_on_previous_text=False` в обоих вызовах `model.transcribe()` |
| Вырожденные word-level таймкоды | Изредка faster-whisper отдаёт всем словам длинного сегмента ОДИНАКОВЫЙ таймкод — реплика с `start==end` никогда не подсвечивается при воспроизведении | Границы самого сегмента (из спецтокенов таймстемпов в основном цикле генерации) остаются надёжными даже когда word-level DTW-выравнивание внутри сегмента схлопывается — при обнаружении вырождения слова раскидываются по границам сегмента пропорционально длине слова |
| Длинные записи не помещаются в GPU-память целиком на ASR | Whisper по умолчанию обрабатывает файл целиком | Чанкование ASR по паузам (см. выше), диаризация не чанкуется вовсе |
| Потеря задачи при обрыве соединения с Redis | `BLPOP` удаляет элемент на сервере атомарно ещё до получения ответа клиентом — обрыв в этот момент теряет задачу бесследно | `BLMOVE` в отдельный `processing`-список; при старте `worker` первым делом забирает оттуда всё, что осталось от предыдущего (упавшего) запуска, и возвращает в начало очереди |
| Падение `worker` посреди GPU-инференса | Нативный CUDA-краш не ловится Python-исключением, процесс воркера падает целиком | `backend` и `worker` — раздельные контейнеры без общей памяти; `docker-compose` политика `restart: unless-stopped` поднимает `worker` заново, `backend` в это время продолжает отвечать. При старте `worker` помечает `error` задачи, застрявшие в `processing` (не `queued` — те целы в Redis) |
| Видео на вход | Пользователь загружает `.mp4`, а не аудио | `ffmpeg -vn` на препроцессинге вырезает только аудиодорожку, видеопоток отбрасывается сразу |

## Метрики

Измерено на реальном сервере (RTX A4000), тестовая запись — лекция, 1798с (~30 мин):

| | ASR-движок | RTF (общий) | Время на 30-мин запись |
|---|---|---|---|
| Было | openai-whisper (PyTorch) | 0.339 | ~10 мин |
| Стало | faster-whisper (CTranslate2), с фиксом от зацикливания | 0.201 | ~6 мин |

RTF (real-time factor) = время обработки / длительность записи; меньше — быстрее. Итог — обработка примерно в **1,7 раза быстрее**, при этом корректная (без повторов).

Качество (на эталонном транскрипте `audio/script.pdf`, `pipeline/evaluate.py`):
- **WER 18%** — на реальной, не идеально чистой записи.
- **Точность определения ролей 99.6%** — несмотря на кейс с неравным объёмом речи (H2 выше).

## Запуск

```bash
cd app
docker compose up --build
# http://<host>:11111
```

Перед первым запуском стоит проверить, что GPU реально проброшена в Docker:
```bash
docker run --rm --gpus all nvidia/cuda:12.1.0-base-ubuntu22.04 nvidia-smi
```

Есть и CLI для разового прогона без сервиса — `python -m pipeline.cli audio.mp3 --reference script.pdf` (из `app/`, после `pip install -r requirements-worker.txt`).

API: `POST /api/upload` → `POST /api/transcribe/{file_id}` → опрос `GET /api/status/{file_id}` (стадия, RTF-метрики) → `GET /api/transcript/{file_id}`. `GET /api/jobs?limit=&offset=` — список задач для UI.

## Известные ограничения

- Роль — ручная эвристика, веса скоринга подобраны вручную, не откалиброваны на датасете.
- NeMo — CUDA/CPU only, Apple MPS не поддерживает.
- Диаризация всегда одним проходом на весь файл — на записи, не влезающей в GPU-память целиком, это станет следующим узким местом (ASR к тому моменту уже решается чанкованием).
- `worker` разбирает очередь строго по одной задаче — Whisper и NeMo на одной карте не терпят параллельного инференса.
